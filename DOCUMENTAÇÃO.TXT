INTRODUÇÃO -------------------------------------------------------
BOT UTILIZADA ESTRUTURA MVC: DENTRO DA PASTA MVC CONTEM OS SEGUINTE PASTA:

CONTROLLER :
responsavel por ligar camada model, contem os seguintes arquivos:
    -controller.comissao_spf.py   =  responsavel pela LIQUIDAÇÃO comissao_spf;
    -controller.plus_bancario.py  =  responsavel peloa LIQUIDAÇÃO plus_bancario;
    -controller.plus_posterior.py =  responsavel  pela LIQUIDAÇÃO plus_posterior.

MODELL :
responsavel pela logica do rpa, ela se divide em dois tipos, primeiro são arquivos globais e pasta de cada liquidaçõa
    global:
    arquivos py global são chamando pois são arquivos que podem ser usando dentro das tres tipos de função de liquidacão
    pastas:
    Existem tres pasta dentro do model: Comissao_spf, plus_bancario, plus_posterior, cada pasta contem modulos com seus
    respectivos função

VIEW:
    Durante a escrita desse codigo a pasta view era so de enfeite.

O ARQUIVO MAIN SE CHMA RPA_LIQUIDACAO_COMISSAO.PY E FICA LOCALIZADADO FORA DA ESTRUTURA MVC É NELE ONDE VAI SER CHAMADO AS FUNCOES DO DENTRO DA PASTA
CONTROLLER SÃO PLANEJADAS E ORGANIZADAS COM FLUXO DO CODIGO 

FUNCIONAMENTO----------------------------------------------------------
NO ARQUIVO MAIN VC VAI ENCONTRA CONEXÃO COM PLANILHA A LEITURA DA PLANILHA E AS VERIFIÇOES E AS TRES CHAMADAS DO CONTROLLER E ATUALIZAÇÃO DA PLANILHA CASO LIQUIDACAO SEJA FEITAS
 - PRIMEIRA VERIFICAO :
        QUAIS LINHAS AINDA FALTAM SER FEITA
 - SEGUNDA VERIFICAO : 
        QUAL TIPO DE BONIFICÃO DE ACORCO COM BONIFICÃO VAI SER CHAMADA UM CONTROLLER DE SUA RESPECTIVA BONIFICÃO

EXPLICAÇOES ---------------------------------------------------------------------------------------
PLUS BANCARIO:'
PRIMEIRO PASSO É A IDENTIFIÇÃO DO VALOR NO CONTROLE BANCARIO;
    ACESSA SISTEMA DO CONTROLE BANCARIO DE ACORDO COM EMPRESA QUE VEM DA PLANILHA, DEPOIS REALIZA UM CALCULO PARA PESQUISA VALOR,
    FORMULA DO CALCULO É ("Valor Total da Nota Fiscal" * 0,985), BOT PEGAR VALOR DA COLUNA 'Valor Total da Nota Fiscal' EM VEM PLANILHA PROCURA ESSA VALOR
    CASO ELE ACHA VALOR VAI PARA PROXIMO PASSO CASO ELE N ENCONTRE VALOR PULA PARA PROXIMA LINHA.

SEGUNDO PASSO IDENTIFIÇÃO DOS CLIENTES NO DEALER:
    QUANDO VALOR FOR ENCONTRA BOT IRA PEGAR SALVA OS NOMES DO CLIENTE QUE CORRESPONDEM A LINHA DO PROCESSO, DEPOIS IRA FAZER CONSULTA NO CONTAS A RECEBER PARA VERIFICAR SE CLIENTE ENCONTRA CADASTRADO
    CASO CLIENTE SEJA ENCONTRA BOT VAI SALVA INFORMAÇÃO DO CLIENTE CASO N ENCONTRE TBM IRA SALVA INFORMAÇÃO DO CLIENTE

TERCEIRO PASSO PESQUISA PELO TITULO:
    BOT VAI PESQUISA PELO TITULO DO CLIENTE NO SISTEMA DO CONTA A RECEBER CASO ENCONTRE TITULO VAI PARA PROXIMO PASSO 
    CASO N ENCONTRE TITULO SALVA AS INFORMAÇÕES COMO IDENTIFIÇÃO DE TITULO NÃO ENCONTRADO VAI PARA PROXIMO CLIENTE.

QUARTO PASSO REALIZA LIQUIDACAO:
    COM TITULO ENCONTRADO E COM SISTEMA JA ABERTO DO CONTAS A RECEBER BOT COMEÇA REALIZAR LIQUIDACAO DO TITULO 
    COM LIQUIDACAO REALIZADA BOT SALVA AS INFORMAÇÃO COM TITULO DE LIQUIDACAO FEITA E SALVA AS INFORMAÇÕES DEPOIS VAI PARA PROXIMO

EXCESSOES:
 PRIMEIRA EXCEÇÃO VALOR DIVERGENTE:
    PODE ACONTECE QUE VALOR SEJA ENCONTRA E TITULO TBM ENCONTRA MAS VALOR QUE VEM DO DEALER DIVERGENTE DO VALOR DA PLANILHA
    NESSA CASO INFORMAÇÃO DO CLIENTE SERA SALVA COM TITULO DE VALOR DIVERGENTE
 SEGUNDA EXCEÇÃO TITULO N ENCONTRADO:
  CASO TITULO N SEJA ENCONTRA SERA SALVA AS INFORMAÇÃO DO CLIENTE COM TITULO DE NÃO ENCONTRA
  TERCEIRO EXCEÇÃO EMPRESA DIVERGENTE MAS TITULO N ENCONTRA:
    PODE ACONTECE QUE EMPRESA SEJA DIVERGENTE MAS N ENCONTRA TITULO
    QUARTA EXCEÇÃO EMPRESA IGUALS:
        EMPRESA PODEM SER IGUAL MAIS TITULO N ENCONTRA
